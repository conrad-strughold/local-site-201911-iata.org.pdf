Migration to New MCT Standard 

 

You  will  find  below  the  necessary  steps  to  take  for  your  migration  to  the  new  MCT 
standard. This check-list includes the testing activities and target migration milestone for 
industry adoption. 

Checklist: 



  Review internally the Awareness deck and Executive deck (which can be found on 

the SISC site 

www.iata.org/sisc

 ) 



necessary 



  Appoint a MCT project Manager 



  Talk to your system providers working with MCT data  



  Contact your codeshare partners to raise their awareness of the issue and clarify 

if  you  can  accept  the  MCTs  that  they  will  file  for  their  operating  flights  for  your 
marketing flights

  Contact your Data Aggregators

Tips for Cleaning Up Your MCT file 

 

1.  Within your itinerary cross check: 



  operation/mkt, Example:  if you have an DOM DOM MCT at JFK make sure you 

have domestic flight that arrive/depart JFK 



  terminal, make sure MCTs filed with terminals actually fly to those terminals 



  suppression if you publish with traffic restrictions, as some MCTs might be 

unnecessary  



  flight number ranges, such as the ones with one flight number and make sure 

you actually fly that route with that number 
 

2.  Isolate codeshare ranges and if default to operating, just have those MCTs mapped out 

when the time comes 
 

3.  Check for redundant information, such as: 



  if you have a waver, make sure it’s different from airport default; if the same then 

delete MCT   



  or if you have 2 equivalent wavers and the more specific one is forcing the same 

rule as the less specific one, then deleted the more specific one. 

 

4.  Make the data match between aggregators 



  Take one of the aggregator files (the easiest to work with) and ask the other 

aggregator to do the corresponding changes to have the same info as the file you 
sent them. Make sure you feel comfortable with the file you are sending them. 



  Compare both files and send corresponding changes to each aggregator to make 

the info equivalent.  

 

5.  When cleaning files and when you are not the outgoing airline, add a comment on the 

MCT you need to delete explaining why it is not necessary anymore, this way the 
outgoing airline will not have to question why you want to delete this record. The 
outgoing airline can then go straight into checking if you are right and concur faster.  

 

Best Practices Going Forward 

Ask yourself the question: Do I have less exceptions if I change the standard MCT? 

Analyze reason of exception and file exceptions according to reason 

Try to use a broader MCT as much as possible rather than file multiple specific MCTs  

Use discontinued dates on wavers  

Consider the global and regional suppressions where possible

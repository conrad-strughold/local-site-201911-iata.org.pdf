1.

 

Introduction to the SIS Change Management Process 

All enhancements to SIS are subject to the SIS Change Management Process.  SIS enhancements 
can be requested by any SIS member via  the 

Customer Portal 

by selecting the 

Topic as “SIS –

 

Si

mplified Invoicing & Settlement”

 and the Subtopic 

as “New Feature

 Request

”

. 

There  are  two  SIS  Major  Releases  scheduled  per  year.    The  SIS  Steering  Group  reviews  and 
prioritizes the enhancements to be included in each Major Release. 

2.

 

Open Change Requests in SIS 

The following table lists all open Change Requests raised by members.  The “Requirement Status” 
column shows the status of the requirements.  A requirement status of “Pending” means the item 

is still under discussion with the vendor or relevant Working Group. 

Requirement status key: 

•

 

Requirements finalized 

–

 Requirements finalized, will be implemented in a future release 

•

 

Requirements in progress 

–

 In discussion with the vendor / Working Group 

•

 

On Hold 

–

 Relevant working group opted to put the change request On Hold for an indefinite 

period 

•

 

Pending WG Review 

–

 Awaiting discussion with the relevant Working Group

710 

Future  Submission  of 

MISC  Invoices  with  or 

without 

Future 

Exchange Rates 

MISC 

Requirements 

finalized 

Currently the system expects an exchange rate to be 

present  in  the  master  for  tolerances  and  MIN/MAX 

conversions.  Therefore,  if  the  member  would  like  to 

submit an invoice for a future month, where the rate is 

not available then it would fail for tolerance. 

 

This change will use the current month exchange rate 

to do the tolerance and MIN/MAX calculations. 

712 

Block for Bilateral 

Billings 

MISC 

Requirements 

finalized 

Currently, it is not possible to define ‘Blocking Rules’ 

for Bilateral Billings.  

This  change  will  allow  SIS  members  to  benefit  of  a 

blocking mechanism for bilateral invoices. Similarly to 

the  existing  mechanism,  in  the  clearing  houses,  a 

member can be blocked from submitting or receiving 

invoices.  This  blocking  will  be  handled  by  the  SIS 

Operations team. 

741 

Purging of Invoice 

PDFs and Listings 

based on Payment 

Status 

MISC 

Requirements 

finalized 

Currently, Invoice PDF, Listing and CSV listings files of 

Bilateral MISC Invoices/Credit Notes are purged after 

12  months.  With  this  CMP,  invoices  marked  as 

Overdue by the Billing Member will be excluded from 

the purging process and will remain available to view 

and download to the Billed Member until the Payment 

Status is updated to a different value. 

 

This CMP is selected for release 1.15 in Nov. 2019

768 

Hard validation of 

Double Billings for 

First Billings 

PAX 

Requirements 

finalized 

Currently,  some  airlines  are  issuing  multiple  first 

billings  for  the  same  coupon,  such  as  rebilled 

documents  from  Form  C  (SC51)  billed  again  in  Non-

Sampling with SC01/02/03 or simply multiple SC01 or 

SC08  for  the  same  coupon  in  the  same  or  different 

invoices. SIS provides a validation of these cases, but 

the coupon positions are only flagged as ‘DU’ / Double 

Use (IS Validation Flag) and they do not fail validation. 

With this change a hard validation will be implemented 

for  multiple  first  billings  having  the  same  Issuing 

Airline,  Document  &  Coupon  Number  as  well  as  the 

same Billing Airline if the previous first billing has been 

raised  not  more  than  12  months  in  advance  (due  to 

reuse of document numbers). 

No  validation  error  shall  occur  in  case  the  following 

conditions are fulfilled: 

- the preceding first billing was on a Form A 

- the billed airline did put the coupon on Form C 

- the second first billing has SC27/90/95/96 

If airlines wish to send additional billings on top of the 

existing first billing already raised, it should only take 

place with the prior approval of the billed airline and 

should be billed through Source Code 09. 

771 

Multiple files handling 

for carriers with full 

file loading 

ALL 

Requirements 

finalized 

Currently,  carriers  opting  to  control  the  validation 

process at the file level (“reject file in error”) must log 

in to SIS to delete the failed file before being able to 

resubmit the corrected file. The need for accessing IS-

WEB requires extra training and extra time to log in and 

delete  the  file.  Since  users  with  that  authority  can 

delete all files, it represents a security issue. 

With this change, for carriers that selected to control 

the validation process at the file level, when a file will 

fail validation, it will be automatically deleted after the 

validation report is generated. This way, the carrier can 

correct  the  issue  and  upload  the  file  again  without 

having to involve an authorized user with the proper 

training to have the file deleted. 

778  Deletion of user data 

ALL 

Requirements 

finalized 

Following  the  changes  in  the  EU  legislation  with 

regards to the storage of personal data, a new process 

is  introduced  to  automatically  invalidate  and  delete 

details of users who have been deactivated by other 

users or deactivated by the system for inactivity.
